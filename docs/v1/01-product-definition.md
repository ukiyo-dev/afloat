# 产品定义

Afloat 是一个个人承诺观察系统。它以日历作为唯一时间入口，读取用户写下的承诺与偏移记录，生成时间事实镜像。

一句话定义：

> Afloat 不安排你的时间，只呈现承诺如何成为事实，或如何偏离事实。

## Afloat 解决的问题

后现代环境中，娱乐的启动成本极低，而理想的短期预期往往为负。人会在清醒时懊悔，又在选择时沉沦。

Afloat 不试图用意志力解决这个问题。它引入一种轻量仪式：让过去的我写下承诺，让未来的我无法假装时间没有流向某处。

## 产品姿态

Afloat 是镜子，不是教练。

它不说用户应该做什么，不评价行为好坏，也不提供外部刺激。它只呈现：

- 用户承诺了什么
- 承诺是否被时间承载
- 时间最终兑现、偏移或空白在哪里
- 哪些主题正在发热、冷却或沉寂
- 哪些自我约束仍然有效，哪些约束已经被打破

## 内在凝视

Afloat 承认一个风险：用户自愿构造的承诺系统，仍然可能反过来压迫用户。

计划、规则、公开镜像和统计口径都可以由用户自己选择，但自愿不等于无压迫。用户可能把承诺账本内化为新的审判者，用兑现率、偏移记录或连续守约天数证明自己是否足够自律、认真或配得上休息。

因此 Afloat 不把用户自建的秩序神圣化。系统允许用户观察自己如何管理自己，也必须让这种管理本身保持可见、可怀疑、可停用。

Afloat 的事实镜像只能描述结构，不能升级为身份判断。它应说：

- 这段承诺被兑现
- 这段承诺被内部偏移覆盖
- 这个线程近期承载减少
- 未承诺时间增加

它不应说：

- 用户失败了
- 用户缺乏自律
- 用户浪费了时间
- 用户战胜了娱乐

如果 Afloat 让用户更诚实，它是镜子。如果它让用户更羞耻，它就变成了压迫。

## 欲望与现实

Afloat 不把工作理解为压迫，也不把娱乐理解为解放。

工作可能是对他者要求的服从，也可能是对自身欲望的忠实。娱乐可能是滋养性的奖赏，也可能是对欲望的逃避。计划类型不是道德类型，也不是欲望类型。

Afloat 不自动判断某段时间是否忠实于欲望，也不替用户解释动机。系统只能呈现承诺、偏移与事实的结构；欲望关系必须由用户自己在解释层中命名。

看清欲望并不总是令人轻松。欲望会带来代价：时间代价、关系代价、身份代价，以及放弃其他可能性的代价。正视现实也可能意味着承认某些承诺并未真正承载欲望，某些娱乐只是遮蔽，某些工作反而不可放弃。

因此 Afloat 的目的不是让用户更轻松地优化自己，而是提供一个足够冷静的事实镜像，使用户有机会看见欲望如何被承诺、被兑现、被回避，或被现实改写。

## 渐进使用

Afloat 不要求用户一次性建立完整协议。

用户可以只从一个语义日历开始，例如只记录睡眠、只记录娱乐，或只记录理想。系统应当从已有记录生成局部镜像，而不是因为协议不完整而阻断使用。

完整协议是长期形态，不是启动门槛。

## 不是任务管理

Afloat 不创建任务，不管理 checklist，不判断完成失败。

线程不是任务，而是可持续追踪的主题容器。线程可以承载主题承诺，也可以承载 `To Do`、`Relax List`、`Explore List` 这类开放分类。Deadline 不是催促工具，而是承诺型线程的时间边界。可行性分析不是预测用户能否做到，而是呈现当前未来计划是否足以承载承诺。

## 不是习惯养成

Afloat 可以记录用户刚性规则，但规则不是柔性习惯系统。

规则没有补签、豁免、轻微违约或事后补救。违约会结束当前 run，系统只记录违约日期、场景和原因。

## 不是效率工具

Afloat 不追求更多产出，也不要求用户减少娱乐。

娱乐是计划层中的正当时间承诺。Afloat 区分的是主动娱乐和偏移中的娱乐，而不是把娱乐本身视作错误。

## 不是社交产品

Afloat 可以有公开镜像，但公开镜像不是社交关系。

镜像页的访客模式复用本人模式界面，但不展示 Threads 模块。它让用户知道自己的时间结构有一个低强度、非互动、匿名的可见面。

## 硬边界

v1 明确不做：

- 不修改任何外部日历数据
- 不发送通知、提醒或邮件
- 不自动排程
- 不创建 checklist
- 不做评分、排名或柔性连续打卡
- 不提供规则补救、豁免或复活 run
- 不做社交关系
- 不使用 AI 自动解释动机
- 不自动纠错用户协议错误

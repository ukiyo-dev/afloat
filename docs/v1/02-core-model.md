# 核心模型

Afloat 的核心是承诺与事实之间的关系。

整体链路：

```text
线程追踪
  ↓
时间承诺
  ↓
偏移覆盖
  ↓
事实层
  ↓
派生视图

用户规则
  ↓
违约记录
  ↓
规则账本
```

## 线程追踪

线程追踪回答：

> 哪些事情不应该无声无息地从我的生活中消失？

线程是可持续追踪的主题容器。线程由 `Group / Item` 定位，可以由用户主动创建，也可以由日历事件自动聚类而来。

线程可以是承诺，也可以是分类。例如 `To Do`、`Relax List`、`Explore List` 可以作为 Group 存在，用来追踪开放列表中的 Item。

线程可以带：

- `Group`
- `Item`
- `expectedMinutes`
- `start`
- `deadline`

线程不安排时间。它只声明某个主题、分类或事项应该被观察。

`Item` 是重要但不紧急的承诺单位。它适合那些需要跨时间投入、容易被短期现实挤掉、但重要到不能长期漂浮的事项。紧急事项应优先进入日历计划层，被安排到具体时间窗口；不重要事项不应进入线程追踪。

## 时间承诺

时间承诺回答：

> 我把某段时间承诺给了什么？

计划层承载时间承诺，包括三种时间：

- 理想
- 娱乐
- 休息

理想、娱乐、休息都是正当计划。Afloat 不把娱乐视为错误，也不把休息视为无效时间。

三种计划时间继承同一套计划事件结构。它们都可以使用标题协议并进入线程追踪。带 `expectedMinutes` 或 `deadline` 的线程进入承诺可行性分析；`start` 与 `deadline` 使用日期级闭区间定义观察窗口。没有这些目标参数的线程只作为开放追踪项显示。

继承同一结构不代表语义可以混用。

休息类型的承诺必须以恢复为目的。睡眠、午休、放空、恢复性散步可以是休息；如果散步的主要目的变成构思、产出或推进某个主题，它更接近理想，而不是休息。

## 偏移记录

偏移记录回答：

> 这段承诺如何被改写？

偏移层包括两种偏移：

- 外部偏移：原有时间承诺因外部现实而改变
- 内部偏移：原有时间承诺因自身选择而改变

偏移不是第四、第五种普通计划时间。偏移层用于覆盖计划层，生成事实层。

## 事实层

事实层回答：

> 时间最终实际流向了哪里？

事实层不是用户直接输入的数据，而是 Afloat 从计划层与偏移层计算得出的结果。

事实层保留五种结果：

- 理想兑现
- 娱乐兑现
- 休息兑现
- 外部偏移
- 内部偏移

偏移中的娱乐不并入主动娱乐。Afloat 关心的是承诺结构，因此主动娱乐和内部偏移必须分开。

## 派生视图

派生视图是结构化展示数据，不是事实源。

Afloat 每次更新后从事实源全量重算派生视图。线程视图、统计结果、错误列表和访客模式镜像都属于派生视图内容。

Thread 派生视图还可以生成保留 Item `---`：它把已有正式 Thread Group 下、使用明确 `Group：Item` 结构但不进入序号追踪的活动汇总为“不追踪”活动，包括历史事实和信息性 Future。`---` 不是线程声明，不写入事实源，也不具有 Target、Start、Deadline 或可行性状态。

事实源只有：

- 外部日历事件缓存
- 日历语义映射
- 线程声明
- 用户规则
- 规则违约记录
- 每日笔记
- 公开设置

派生视图必须可以随时从事实源重新生成。

## 用户规则

用户规则是用户给自己的限制，具有 `TEST` 与 `SIGNED` 两种承诺状态。`TEST` 是对可能失败的限制进行验证，`SIGNED` 是正式接受该限制；两者共用同一个规则对象、违约历史和 run 模型。

它回答：

> 哪条底线仍然有效？如果被打破，发生在什么场景，原因是什么？

用户规则不同于系统解析规则。系统解析规则用于从日历生成事实层；用户规则用于记录自我约束是否仍然有效。

用户规则包含规则内容、开始日期、承诺状态、启停状态和违约历史。连续守约天数不作为每日打卡长期写入，而是根据开始日期、最近一次违约日期和用户本地今天计算。

两种承诺状态下的违约都会结束当前 run。违约记录固化发生时的类型：`test_break` 或 `rule_break`。从 `TEST` 签署为 `SIGNED` 时，正式规则直接复用签署当下的 current run 起点和天数，run 跨越签署点连续增长；签署也不回溯改变已有违约类型。Overview 的范围履约指标只由正式的 `rule_break` 排除未履约规则。

## 每日笔记

每日笔记是解释层，不是日历协议的一部分。

它回答：

> 用户如何解释当天的承诺、偏移与事实？

每日笔记存储在 Afloat 数据库中，而不是外部日历中。日历负责承诺与偏移，笔记负责反思与解释。

同一个日期允许存在多篇每日笔记。笔记以自身 ID 标识，日期用于归入当天，不作为唯一键。

每日笔记不参与：

- 计划层
- 偏移层
- 事实层
- 承诺兑现率
- 线程统计
- 可行性分析

每日笔记可以设为私密或公开。v1 访客模式复用同一镜像界面，隐藏 Threads 和 Rules 模块；笔记可见性后续可以继续细化。

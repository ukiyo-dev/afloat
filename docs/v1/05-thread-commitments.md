# 线程追踪与可行性

线程是可持续追踪的主题容器。

它回答：

> 哪些主题不应该无声无息地从我的生活中消失？

线程可以是承诺，也可以是分类，例如 `To Do`、`Relax List`、`Explore List`。

线程不是 checklist 系统，也不强制代表完成状态。带 `expectedMinutes` 或 `deadline` 的线程进入承诺可行性分析；没有这些目标参数的线程只作为开放追踪项显示。

## 线程来源

线程有两种来源：

- 主动创建：用户在 Afloat 中声明一个追踪项
- 自动发现：Afloat 从带序号的计划层事件中聚类发现

主动创建的线程即使没有事实推进，也显示为未启动。

自动发现从计划层事件产生。理想、娱乐、休息三种计划类型都可以进入线程追踪。

前端 Threads 模块默认展示活跃线程，并提供 `Active / Inactive / All` 视图过滤。

历史无序号事件会收束同名 `Group / Item` 的当前实例。被历史无序号事件收束的 Item 从 `Active` 视图移入 `Inactive` 视图；同一 Group 下其他未收束 Item 继续显示。

未来无序号事件表示计划中的收束，Item 仍显示在活跃视图中，并标记为将被收束。

## 线程身份

线程由 `Group / Item` 定位。

`Item` 是承诺参数的最小单位。`expectedMinutes` 和 `deadline` 都下发到 `Item`，不直接挂在 `Group` 上。

`Group` 是聚合视图：

- Group 的 `expectedMinutes` = 其下所有 Item 的 `expectedMinutes` 之和
- Group 的 `deadline` = 其下所有 Item 的最晚 deadline
- Group 的已兑现、未来承诺、事实缺口、未计划缺口都由 Item 汇总得到

理想、娱乐、休息可以混合同一个 `Group / Item`。线程是主题容器，三者只是不同性质的时间投入。

示例：

```text
理想：游戏开发：战斗系统 1
娱乐：游戏开发：玩同类游戏 1
休息：游戏开发：恢复散步 1
理想：游戏开发：战斗系统 2
```

这些事件属于同一个追踪主题。

休息线程必须仍然是恢复性追踪项。它可以与某个主题有关，例如围绕长期创作安排恢复，但它的主要目的必须是恢复，而不是构思或产出。

## 标题解析

```text
主标题：副标题 1
```

- `Group = 主标题`
- `Item = 副标题`
- `Sequence = 1`

没有中文冒号时：

```text
写作 1
```

解析为：

```text
Group = 写作
Item = 写作
Sequence = 1
```

质量标记 `🌟` 与 `❔` 不参与线程匹配。

## 序号规则

带序号事件可以开启或延续线程实例。

无序号事件可以收束已有线程实例。

规则：

- `写作 1` 开启线程实例
- `写作 2` 延续线程实例
- `写作` 收束已有线程实例
- 只有 `写作`，不自动生成线程
- 中间缺号不报错，只按时间推进
- 序号倒退是协议错误，但不阻断其他统计

历史无序号事件是同名 `Group / Item` 的实例边界，也重置该 Item 的序号检查。收束后再次出现 `1` 表示新的 Item 实例，不构成序号倒退。

示例：

```text
健康：牙医 1
健康：牙医
健康：牙医 1
```

这表示 `牙医` 被收束后重新开启新实例。

以下才是序号倒退：

```text
健康：牙医 1
健康：牙医 2
健康：牙医 1
```

无数字收束是派生判断，不是永久状态。用户修改历史标题后，派生视图全量重算，线程状态随之改变。

## 删除空线程

用户只能删除空线程。

空线程指：

- 由用户主动创建
- 尚未匹配任何过去事实兑现
- 尚未匹配任何未来计划
- 尚未产生任何自动发现实例

删除空线程表示用户撤回一个尚未进入时间的追踪项。

一旦线程已经有事实历史或未来计划，Afloat 不允许删除它。历史承诺应当保留为事实的一部分。

有历史的线程只能通过无序号事件收束当前实例。

示例：

```text
写作：小说 1
写作：小说 2
写作：小说
```

最后一个无序号事件表示当前实例收束。收束后的 Item 从活跃线程视图移除，但不删除事实历史。

如果未来再次出现：

```text
写作：小说 1
```

则开启新的线程实例。

## 线程统计

线程统计基于事实层，而不是原始计划层。

对于同一 `Group / Item`：

- 已兑现：事实层中的理想兑现 + 娱乐兑现 + 休息兑现
- 未来承诺：当前时刻之后的理想计划 + 娱乐计划 + 休息计划
- 外部偏移：覆盖该线程计划时间的外部偏移
- 内部偏移：覆盖该线程计划时间的内部偏移

被偏移覆盖的计划时间不计入已兑现。

## Deadline 与 expectedMinutes

Item 声明可以带：

- `expectedMinutes`：预期投入时长
- `deadline`：承诺时间边界

它们用于承诺可行性分析。

`expectedMinutes` 只由事实层中的已兑现时间抵扣。未来计划不抵扣，只用于预测未来是否已经承载该承诺。

用户可以在已有 Group 下独立添加 Item。添加 Item 不要求该 Item 已经在日历中出现；主动创建的 Item 即使没有事实推进，也应显示为未启动或未追踪。

Group 不直接编辑 expectedMinutes 或 deadline。若需要改变 Group 聚合结果，用户应编辑其下 Item。

## 可行性指标

核心指标：

```text
已兑现 = 事实层中该线程的理想+娱乐+休息兑现时间
事实缺口 = max(0, expectedMinutes - 已兑现)
未来承诺 = 当前时刻之后该线程的理想+娱乐+休息计划时间
未计划缺口 = max(0, expectedMinutes - 已兑现 - 未来承诺)
计划覆盖率 = 未来承诺 / 事实缺口
```

事实缺口为 0 只表示投入时间已经覆盖目标，不直接表示线程已兑现。

Item 的当前活跃状态由同 `Group / Item` 的带序号与无序号活动决定。当无序号活动已经成为历史时，该 Item 当前实例被收束，并从 `Active` 视图移入 `Inactive` 视图。事实历史仍保留，`Inactive` 视图按同名 Item 汇总所有已收束实例的历史。

如果已兑现但已兑现时间小于 expectedMinutes，历史视图应显示低估比例，表示原计划完成时间被低估。

这些指标先在 Item 级计算，再汇总为 Group 视图。Group 视图用于看主题整体压力，Item 视图用于看具体承诺项。

如果存在 deadline：

```text
每日需安排 = 未计划缺口 / deadline 前剩余天数
```

## 可行性状态

可行性状态只描述承诺结构，不评价用户。

- 已兑现：该 Item 已被同名无序号历史活动关闭
- 已安排：事实缺口大于 0，且未计划缺口为 0
- 仍需安排：存在未计划缺口，但节奏仍可承载
- 节奏紧张：每日需安排接近近期日均承载
- 结构失衡：每日需安排超过近期日均承载
- 已过期：deadline 已过，且事实缺口大于 0
- 需要关注：Active Item 的最近事实活动距离当前时间超过设置阈值，空 Item 使用创建时间

近期日均承载默认使用最近 30 天计划层兑现的日均时长，即理想兑现 + 娱乐兑现 + 休息兑现。

需要关注阈值默认为 7 天，可在设置页调整。该状态由客户端按当前时间计算，不写回事实源或派生视图。

页面展示数字优先于评价：

```text
目标投入：60h
已兑现：20h
事实缺口：40h
未来承诺：15h
未计划缺口：25h
```

## 不做

线程 v1 不做：

- 完成按钮
- 子任务
- checklist
- 自动排程
- 到期提醒
- 逾期惩罚
- AI 建议
- 模糊匹配

## 当前 UI 行为

Threads 模块默认展示 `Active` 线程。`New Plan` 旁提供 `Active / Inactive / All` 过滤和搜索框，用于过滤当前视图中的 Group / Item。

线程列表默认按 deadline 升序排列，没有 deadline 的 Item 排在有 deadline 的 Item 之后。deadline 相同时，再按可行性状态排序。

`Active` 显示当前实例尚未被历史无序号事件收束的 Item。

`Inactive` 显示曾经开启、但当前没有活跃实例的 Item。Inactive Item 的指标按 all-time closed history 汇总，即汇总该同名 Item 的所有已收束实例。

`All` 显示同一 Group 下的 Active 与 Inactive Item。Group 名称是稳定命名空间，状态属于 Item，不属于 Group。

搜索命中 Group 时，显示该 Group 下当前视图包含的 Item。搜索命中 Item 时，只显示匹配的 Item。

`Recent Activities` 不暴露实例概念。UI 用虚线切分未来计划与过去事实，用实线切分由无序号收束形成的历史片段边界。

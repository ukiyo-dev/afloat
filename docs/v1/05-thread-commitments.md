# 线程追踪与可行性

线程是可持续追踪的主题容器。

它回答：

> 哪些主题不应该无声无息地从我的生活中消失？

线程可以是承诺，也可以是分类，例如 `To Do`、`Relax List`、`Explore List`。

线程不是 checklist 系统，也不强制代表完成状态。带 `expectedMinutes` 或 `deadline` 的线程进入承诺可行性分析；`start` 与 `deadline` 共同定义承诺观察窗口。没有这些目标参数的线程只作为开放追踪项显示。

Thread Item 主要维护重要但不紧急的承诺。它不是所有待办事项的容器，而是一个需要跨时间投入、不会被紧急性自然推动、但重要到不能长期漂浮的承诺单位。

适合进入 Item 的事项通常满足：

- 重要到值得持续追踪
- 需要多次时间投入
- 容易被短期事务挤掉
- 如果不追踪，会长期停留在意愿或漂浮状态
- 需要 `expectedMinutes` 或 `deadline` 来暴露承诺结构

不适合进入 Item 的事项包括：

- 单次紧急任务
- 当天待办
- 可以立刻完成的小事
- 低价值杂事
- 纯提醒性质的事项

紧急事项应优先进入日历计划层，由用户安排到具体时间窗口；如果执行中被现实改写，再由偏移层记录。Threads 只在事项需要跨时间追踪、且不会被紧急性自然推动时介入。不重要事项不应进入 Afloat，否则 Threads 会退化为 checklist。

## 线程来源

线程有两种来源：

- 主动创建：用户在 Afloat 中声明一个追踪项
- 自动发现：Afloat 从带序号的计划层事件中聚类发现

主动创建的线程即使没有事实推进，也显示为未启动。

自动发现从计划层事件产生。理想、娱乐、休息三种计划类型都可以进入线程追踪。

前端 Threads 模块默认展示活跃线程，并提供 `Active / Inactive / All` 视图过滤。

历史无序号事件会收束同名 `Group / Item` 的当前实例。被历史无序号事件收束的 Item 从 `Active` 视图移入 `Inactive` 视图；同一 Group 下其他未收束 Item 继续显示。

未来无序号事件表示计划中的收束，Item 仍显示在活跃视图中，并标记为将被收束。

## 线程身份

线程由 `Group / Item` 定位。

`Item` 是承诺参数的最小单位。`expectedMinutes`、`start` 和 `deadline` 都下发到 `Item`，不直接挂在 `Group` 上。

`Group` 是聚合视图：

- Group 的 `expectedMinutes` = 其下所有 Item 的 `expectedMinutes` 之和
- Group 的 `start` = 其下所有 Item 的最早 start
- Group 的 `deadline` = 其下所有 Item 的最晚 deadline
- Group 的已兑现、未来承诺、事实缺口、未计划缺口都由 Item 汇总得到

理想、娱乐、休息可以混合同一个 `Group / Item`。线程是主题容器，三者只是不同性质的时间投入。

示例：

```text
理想：游戏开发：战斗系统 1
娱乐：游戏开发：玩同类游戏 1
休息：游戏开发：恢复散步 1
理想：游戏开发：战斗系统 2
```

这些事件属于同一个追踪主题。

休息线程必须仍然是恢复性追踪项。它可以与某个主题有关，例如围绕长期创作安排恢复，但它的主要目的必须是恢复，而不是构思或产出。

## 标题解析

```text
主标题：副标题 1
```

- `Group = 主标题`
- `Item = 副标题`
- `Sequence = 1`

没有中文冒号时：

```text
写作 1
```

解析为：

```text
Group = 写作
Item = 写作
Sequence = 1
```

质量标记 `🌟` 与 `❔` 不参与线程匹配。

## 序号规则

带序号事件可以开启或延续线程实例。

无序号事件可以收束已有线程实例。

规则：

- `写作 1` 开启线程实例
- `写作 2` 延续线程实例
- `写作` 收束已有线程实例
- 只有 `写作`，不自动生成同名正式 Thread Item
- 已经出现过带序号事件并被收束后，后续连续出现的同名无序号事件仍归入该已收束实例的事实统计与 Recent Activities；它们不会开启新实例，也不会让 Item 恢复 Active
- 中间缺号不报错，只按时间推进
- 序号倒退是协议错误，但不阻断其他统计

历史无序号事件是同名 `Group / Item` 的实例边界，也重置该 Item 的序号检查。收束后再次出现 `1` 表示新的 Item 实例，不构成序号倒退。

示例：

```text
健康：牙医 1
健康：牙医
健康：牙医 1
```

这表示 `牙医` 被收束后重新开启新实例。

以下才是序号倒退：

```text
健康：牙医 1
健康：牙医 2
健康：牙医 1
```

无数字收束是派生判断，不是永久状态。用户修改历史标题后，派生视图全量重算，线程状态随之改变。

重复无序号终止用于兼容长跨度事项：用户可能已经不记得序号，或只想继续记录活动但不希望事项长期保持 Active。单独出现、此前从未有过同名带序号事件的无序号事件不生成同名正式 Item；若它使用明确 `Group：Item` 结构且该 Group 已有其他正式 Item，则可以按下节规则汇入 `---`。

如果 Item 已被历史无序号事件收束，后续同名无序号未来计划仍累积到该 Inactive Item 的 Future，并显示在 Recent Activities 的未来区域。这些时间是信息性计划：不重新激活 Item，也不恢复 Target、Deadline、缺口或可行性计算。只有新的同名带序号事件才能开启新实例并恢复承诺计算。

## `---` 不追踪

使用明确 `Group：Item` 结构、且从未进入同名带序号实例的无序号计划活动，按 Group 汇入派生保留 Item `---`。用户也可以直接使用 `Group：---` 显式记录这类活动。

规则：

- `---` 只有 Group 归属和不追踪活动含义，不是线程承诺
- `Group：--- 1` 保留原始标题中的数字，但 Afloat 不将其解析为序号
- 同一 Group 下的不追踪活动统一汇入一个 `---` Item，Recent Activities 继续展示各活动的原始标题
- 连续、同名且时间相邻的不追踪活动仍分别显示，不合并为一条 Recent Activity
- 没有中文冒号、因而按协议得到 `Group = Item` 的单标题活动不进入 `---`
- `---` 只能辅助同 Group 中至少一个正式 Thread Item，不能作为该 Group 唯一的 Item 出现；若 Group 尚无声明或自动发现的正式 Item，这些活动仍保留在事实层，但不生成 Threads Group
- `---` 的事实投入、最近活动和偏移计入 Group 的事实汇总
- `---` 不参与 Group 的 Active 状态、Start、Deadline、Target、缺口、可行性或待规划判断；界面中只显示在 `Inactive` 与 `All` 视图
- `---` 累积并展示同 Group 不追踪活动的 Future，但这些时间只是信息性计划，不产生 Target、Deadline、缺口或可行性语义
- `---` 不能由用户作为普通 Thread Item 创建、编辑或删除

因此，`---` 影响 Group 的“做了什么”，但不影响 Group 的“承诺了什么”。

## 删除空线程

用户只能删除空线程。

空线程指：

- 由用户主动创建
- 尚未匹配任何过去事实兑现
- 尚未匹配任何未来计划
- 尚未产生任何自动发现实例

删除空线程表示用户撤回一个尚未进入时间的追踪项。

一旦线程已经有事实历史或未来计划，Afloat 不允许删除它。历史承诺应当保留为事实的一部分。

有历史的线程只能通过无序号事件收束当前实例。

示例：

```text
写作：小说 1
写作：小说 2
写作：小说
```

最后一个无序号事件表示当前实例收束。收束后的 Item 从活跃线程视图移除，但不删除事实历史。

如果未来再次出现：

```text
写作：小说 1
```

则开启新的线程实例。

## 线程统计

线程统计基于事实层，而不是原始计划层。

对于同一 `Group / Item`：

- 已兑现：事实层中的理想兑现 + 娱乐兑现 + 休息兑现
- 未来承诺：当前时刻之后的理想计划 + 娱乐计划 + 休息计划
- 外部偏移：覆盖该线程计划时间的外部偏移
- 内部偏移：覆盖该线程计划时间的内部偏移

被偏移覆盖的计划时间不计入已兑现。

Recent activities 展示事实层活动本身。覆盖某线程计划的偏移应出现在该线程历史中，但只显示偏移活动标题，例如 `内部偏移：刷手机`，不追加 `覆盖 Afloat：MVP 1` 之类说明。若偏移范围超出该线程计划，只将偏移与计划重叠的区间计入该线程历史。

跨越当前时刻的计划按当前完整分钟拆分：当前分钟之前显示为事实，之后显示为 Future。客户端投影将运行时向下取整到完整分钟，因此不会生成显示为 `15:00-15:00 / 0m` 的瞬时事实片段。对于已收束 Item 的无序号终止尾部以及 `---`，拆分仍保留原计划的真实开始时间，而不是把整个活动错误地显示为从派生时刻开始。

## Start、Deadline 与 expectedMinutes

Item 声明可以带：

- `expectedMinutes`：预期投入时长
- `start`：承诺开始进入观察的日期
- `deadline`：承诺时间边界

它们用于承诺可行性分析。

`deadline` 不是催促工具，也不是为了把紧急事项放进 Threads。它用于给重要但不紧急的承诺设置观察窗口，使 Afloat 能呈现当前未来计划是否真的承载了这项承诺。

`start` 让用户可以提前声明更远未来的承诺。Start 之前 Item 显示为「即将到来」，可以提前安排日历计划，但不进入需要关注判断，也不触发 Overview 的红色待规划状态。Start 当天即进入活跃观察。

Start 与 Deadline 都是用户时区下的日期，不具有时分秒语义。承诺窗口是闭区间：`start <= 当前日期 <= deadline`。Start 与 Deadline 可以是同一天；只有 `start > deadline` 才是无效声明。Deadline 当天仍未过期，次日才进入已过期状态。

线程时间线同样按用户本地日历日处理声明边界。声明的原始 `createdAt` 保留完整时间用于审计，但派生排序和事实窗口起点忽略其时分秒；同一用户本地日内，声明先于该日的计划活动处理，且该日 00:00 起的同名活动都属于该声明。因此，即使同名无序号活动在当天早于声明创建时间开始，它仍可收束并计入该声明的 Item。不同日期之间仍按日期先后处理。

新建 Thread 或在 Group 下添加 Item 时，Start 默认使用用户时区下的当天日期。编辑已有 Item 时回填其有效 Start。历史声明没有显式 Start 时，以 `createdAt` 对应的用户本地日期作为有效 Start，并在编辑表单中显示该日期；编辑不得自动把它改成当天。自动发现且没有声明记录的线程，以首次带序号计划事件的日期作为 Start。

Start 是观察窗口起点，不是事实统计边界。Start 之前已经产生的事实投入和计划仍正常计入线程统计。

`expectedMinutes` 只由事实层中的已兑现时间抵扣。未来计划不抵扣，只用于预测未来是否已经承载该承诺。

### Steady Daily 负载

线程声明可以设置 `steadyDaily`，表示该 Item 虽然允许某天不投入并在之后补偿，但应优先保持剩余窗口中的日投入均匀。Target 仍由 `expectedMinutes` 声明，Daily 始终由事实缺口和剩余窗口派生，不作为固定分钟数持久化。

- `steadyDaily = false`：普通可流动负载，可以承担较多削峰填谷
- `steadyDaily = true`：以 `factGapMinutes / remainingDays` 形成稳定底座，不承担跨区间填谷
- Steady Daily 必须同时具有 Deadline
- Today 的总量受当前剩余窗口平均控制；Item 之间的正负节奏差异可以在 Today 内迁移
- 某天未投入或超额投入，不直接把总负载无限推高或压低，而是改变次日优先推进的 Item
- 历史声明和自动发现 Item 默认 `steadyDaily = false`

`steadyDaily` 是分配偏好，不是每日独立结算协议；`dailyRequiredMinutes` 仍表示系统根据事实缺口计算的当前所需兑现速度。

Daily Load 从用户时区下的当天开始投影，但 Today 必须作为独立结算点，不与未来日期压缩成日均区间。系统先以当前所有 Item 的事实缺口和剩余窗口，计算 Today 的总平均额度；再根据各 Item 的累计节奏余额决定这笔额度应分给谁：

```text
累计应达 = expectedMinutes × 已经过窗口天数 / 完整窗口天数
节奏余额 = max(0, 累计应达 - 已兑现)
Today 总额度 = 总事实缺口 / 当前剩余天数
Today Item 分配 = 按节奏余额比例分配 Today 总额度
```

节奏余额只用于决定 Today 的 Item 组成，不把未来逐日排程展示给用户。例如两个 Item 的窗口日均均为 30 分钟，上一日只推进第一个 Item 60 分钟，则次日第一个 Item 为 0、第二个 Item 为 60 分钟。若两个 Item 都没有推进，Today 总额度会按剩余窗口平均上升，但不会人为偏向某个 Item。Today 结算后，非 Today 只展示剩余事实缺口在未来窗口中的平均值。

Today 采用日初结算语义。系统使用截至上一自然日的事实确定当天总额度和 Item 分配；当天新增事实只从对应 Item 的日初额度中扣减，不在同一天因为仍有历史节奏欠额而重新补满。某 Item 的日初额度被当天事实覆盖后继续以负数显示超前量，例如日初 30 分钟而当天已兑现 50 分钟时显示 `-20m`。负数使用休息语义色，表示已提前消费未来份额、不必继续推进，不表示风险；它只参与 Today 净额展示，不得被加回未来事实缺口。尚未消化的历史欠额留待未来平均重算。跨过自然日后，系统再以新的事实快照生成下一日 Today。

Today 结算后，系统再对剩余事实缺口投影未来负载：先铺入所有 Steady Daily，再将普通可流动 Item 按有效窗口天数升序、Deadline 升序和 Start 升序依次分配。短窗口承诺先稳定落位，长窗口承诺再填充当前低水位日期，使宏观负载趋于均匀，同时避免在等价总负载下反复改写 Item 内部分配。可流动 Item 的分钟总量必须守恒，且不得越过自身 `max(tomorrow, start) ... deadline` 闭区间。未来计划不参与该宏观负载分布。

具有完全相同有效 Start 和 Deadline 的可流动 Items 组成同一个窗口 Cohort。系统先按 Cohort 的 Target 总量进行水位填充，再按各 Item 的 Target 占比分配 Cohort 在每个日期的负载。Thread Key 只用于保证输出顺序稳定，不得决定同窗口 Items 中由谁单独承担填谷。

负载条打开时可以从各 Item 独立均摊产生的原始分布，短暂过渡到全局均衡后的分布；该动画只解释重新分配过程，不表示负载正在实时变化。Steady Daily 底座在过渡中保持不动，并遵循 `prefers-reduced-motion`。

用户可以在已有 Group 下独立添加 Item。添加 Item 不要求该 Item 已经在日历中出现；主动创建的 Item 即使没有事实推进，也应显示为未启动或未追踪。

Group 不直接编辑 expectedMinutes、start 或 deadline。若需要改变 Group 聚合结果，用户应编辑其下 Item。

## 可行性指标

核心指标：

```text
已兑现 = 事实层中该线程的理想+娱乐+休息兑现时间
事实缺口 = max(0, expectedMinutes - 已兑现)
未来承诺 = 当前时刻之后该线程的理想+娱乐+休息计划时间
未计划缺口 = max(0, expectedMinutes - 已兑现 - 未来承诺)
```

事实缺口为 0 只表示投入时间已经覆盖目标，不直接表示线程已兑现。

UI 使用英文短标签：`Target` 表示预计投入，`Done` 表示已兑现，`Plan` 表示当前时刻之后已经写入计划层日历的时间，`Remaining` 表示事实缺口，`Daily` 表示每日需安排。未计划缺口继续作为 Daily 和可行性状态的内部计算量，但不单独展示为指标。v1 也不展示“计划覆盖率”；这两项都会过度强调用户是否已经排完整个长期 Thread，并错误暗示用户应立即安排全部剩余时间。

Item 的当前活跃状态由同 `Group / Item` 的带序号与无序号活动决定。当无序号活动已经成为历史时，该 Item 当前实例被收束，并从 `Active` 视图移入 `Inactive` 视图。事实历史仍保留，`Inactive` 视图按同名 Item 汇总所有已收束实例的历史。

如果已兑现但已兑现时间小于 expectedMinutes，历史视图应显示低估比例，表示原计划完成时间被低估。

这些指标先在 Item 级计算，再汇总为 Group 视图。Group 视图用于看主题整体压力，Item 视图用于看具体承诺项。

如果存在 deadline：

```text
有效起点 = max(当前日期, start)
剩余天数 = deadline - 有效起点 + 1
每日需安排 = 未计划缺口 / 剩余天数
```

当前日期、Start 日期和 Deadline 日期都计入可用天数。

## 可行性状态

可行性状态只描述承诺结构，不评价用户。

- 已兑现：该 Item 已被同名无序号历史活动关闭
- 即将到来：当前日期早于 Start
- 已安排：事实缺口大于 0，且未计划缺口为 0
- 仍需安排：存在未计划缺口，但节奏仍可承载
- 节奏紧张：每日需安排接近近期日均承载
- 结构失衡：每日需安排超过近期日均承载
- 已过期：deadline 已过，且事实缺口大于 0
- 需要关注：已到 Start 的 Active Item，其最近事实活动距离当前日期超过设置阈值；没有事实活动时从 Start 开始计算

近期日均承载默认使用最近 30 天计划层兑现的日均时长，即理想兑现 + 娱乐兑现 + 休息兑现。

需要关注阈值默认为 7 天，可在设置页调整。关注计算只使用用户时区下的日期；Start 或最近事实活动当天计为第 1 个观察日。该状态由客户端按当前日期计算，不写回事实源或派生视图。

页面展示数字优先于评价：

```text
目标投入：60h
已兑现：20h
事实缺口：40h
```

## 当前 UI 行为

Threads 模块默认展示 `Active` 线程。`New Plan` 旁提供 `Active / Inactive / All` 过滤和搜索框，用于过滤当前视图中的 Group / Item。

Item 指标先展示 Progress，Start 紧邻并位于 Deadline 之前；Target、Start、Deadline、Remaining 和 Daily 等未设置或不适用的可行性字段直接隐藏，不显示 `---` 占位。Plan 只在大于 `0` 时显示，Group 汇总遵循同一规则。v1 不单独展示 Unscheduled 或计划覆盖率。存在 Deadline 时，Daily 使用 `每日需安排/闭区间剩余天数` 格式，例如 `30m/60`。Active 列表中，已经到达 Start 的 Item 排在「即将到来」Item 之前：有 Deadline 的 Item 按 Deadline 从旧到新排列；没有 Deadline 的 Item 随后按 Start 从旧到新排列。日期相同时按状态颜色排列，红色状态优先于黄色状态，黄色状态优先于普通状态。「即将到来」Item 按 Start 从近到远排列。

Group 标题不展示「仍需安排」状态标签；该状态保留在 Item 级，Group 只突出更需要整体注意的状态。

Group 摘要右侧右对齐显示 `{已兑现时长} Done / {数量} Items`，Done 与 Items 使用相同字态，让用户在展开前同时看到该 Group 的事实投入和 Item 数量。已兑现时长为 `0` 时省略 Done，只显示 Items 数量。

`Active` 显示当前实例尚未被历史无序号事件收束的 Item。

`Inactive` 显示曾经开启、但当前没有活跃实例的 Item。Inactive Item 的指标按 all-time closed history 汇总，即汇总该同名 Item 的所有已收束实例。

Group 只有在其全部 Item 都处于 Inactive（包括保留 Item `---`）时才显示「已兑现」。即使当前查看的是 `Inactive` 过滤，也必须依据该 Group 的完整 Item 集合判断，不能因为过滤掉 Active Item 而把 Group 标为已兑现。

`Inactive` 是历史查看视图，不显示红色或黄色可行性状态，也不使用红黄边框。若 Group 仍有 Active Item 而不能标记为「已兑现」，在 `Inactive` 中保持无状态展示；风险状态只在 `Active` 与 `All` 中呈现。

`All` 显示同一 Group 下的 Active 与 Inactive Item；同名 Active / Inactive 行合并展示，并以较新的 Start 参与排序。Group 名称是稳定命名空间，Group 标题状态是其完整 Item 集合的派生摘要，不是独立保存的状态。

`Inactive` 与 `All` 视图中的 Group 和 Item 按 Start 从新到旧排序；没有 Start 的记录排在最后。`Active` 继续按可行性与计划边界排序。

搜索命中 Group 时，显示该 Group 下当前视图包含的 Item。搜索命中 Item 时，只显示匹配的 Item。

`Recent Activities` 不暴露实例概念。UI 用虚线切分未来计划与过去事实，用实线切分由无序号收束形成的历史片段边界。连续同名无序号事件属于同一个终止尾部，只在它们与此前带序号活动之间显示一条实线；`---` 不显示实例终止实线。
